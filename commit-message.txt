Add relay server; folder-wiki title/favicon in wiki list; converted-folder saving; collab fixes

Relay server (new: relay-server/)
- Vendored the TiddlyDesktop-to-TiddlyDesktop collaboration relay (Rust / axum
  WebSocket server) into the repo. It only admits OAuth-authenticated users
  (GitHub / GitLab / OIDC), signs a per-connection ES256 membership certificate so
  peers can cryptographically verify each other, and routes only E2E-encrypted
  traffic (it never sees plaintext). Includes per-connection rate limiting, a 60s
  idle timeout with 20s server pings (clients auto-pong), MemberLeft broadcasts when
  a member is reaped, and a grace period that keeps token-protected sync rooms alive
  after the last member leaves. Ships with deploy.sh, setup-server.sh, a systemd unit
  (relay.service), and a .gitignore (excludes target/, relay.db*, and secrets).

Folder wikis & conversion
- Folder windows now report both their title AND favicon to the wiki list. A folder
  wiki runs in its own process (new_instance), so the backstage can't observe its
  DOM the way it does a single-file wiki's iframe; instead the folder window writes a
  small JSON payload {title, faviconType, faviconText} to a per-wiki live-state file
  whenever $:/SiteTitle / $:/SiteSubtitle / $:/favicon.ico change, and the backstage
  fs.watches it (event-driven, no polling). The file is written in place so the
  watcher's inode stays valid, and is cleaned up when a wiki is removed.
- Added the #faviconLink element to wiki-folder-window.html so TiddlyWiki's favicon
  startup has a target (folder wikis boot directly into this page, with no
  template-provided <link>) and the backstage can read the favicon for the list.
- single-file -> folder conversion now adds tiddlywiki/filesystem and
  tiddlywiki/server to the generated tiddlywiki.info (savewikifolder derives it from
  the browser wiki, which has neither), so the converted folder wiki can actually run
  its server and save to disk.

Collaboration plugin
- Reliability (mid-session stalls, sleep/resume): added a client-side liveness
  watchdog. The relay idle-closes after 60s and pings every 20s, but a half-open link
  (sleep, NAT rebind, brief network drop) left us "connected" with all sync dead and
  no reconnect. We now track relay activity (messages + WS ping/pong; the single-file
  bridge forwards "ping"), and force a reconnect when the relay's pings stop arriving
  or the machine clearly slept; an "online" handler reconnects promptly too. A forced
  reconnect re-fires collab-connected, healing coarse + Yjs sync and re-running
  catch-up on both the sleeping and the awake peer.
- Fields/tags committed only at save time now reach co-editing peers: the save-time
  Y.Map flush falls back to the real tiddler when TW's navigator has already saved
  and deleted the draft before our rootWidget handler runs (otherwise it no-oped and
  the change was lost to Yjs).
- Deletions never propagate as deletions, and a subscriber deleting their local copy
  now unsubscribes so a later manifest / reconnect catch-up doesn't silently re-fetch
  and resurrect it.
- Get panel: fixed buttons not rendering at all. [<currentTiddler>get[FIELD]!match[yes]]
  collapses to nothing when FIELD is empty (the get operator drops empty values), so
  every "not yes" branch (subscribe and get-attachment) rendered no button; switched
  to !field:FIELD[yes], which correctly matches empty/absent fields.
- Fixed the External Attachments plugin-title check (external-attachments, not
  externalattachments) in asset-util.js and GetPanel.tid, so storeExternally() and
  the save-as attachment flow actually activate.
- Status dock: pointer-events:auto is now only on the buttons and the open Get/Chat
  panels, not the wrap containers, so the dock no longer covers the wiki content
  underneath; also zeroed the <p> margins that block-mode transclusion injects inside
  the panels (status-bar badges).
- readme: documented the plugins needed for full collaboration (CodeMirror 6 editor,
  $:/plugins/tiddlywiki/codemirror-6-edit-text, $:/plugins/tiddlywiki/external-attachments).
